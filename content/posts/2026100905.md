## 为什么baseurl比API Key更重要？

想象一下，API Key是你家的门钥匙，而baseurl是你家的地址。你光有钥匙没用，得先知道地址在哪，才能走过去开门。绝大多数DeepSeek API接入出错，都是因为baseurl没写对，而不是Key本身的问题。

一个常见的悲剧场景是这样的：开发者从DeepSeek官网拿到API Key，然后直接复制粘贴到代码里：
python
client = OpenAI(api_key="你的DeepSeek-Key")

结果运行时，系统默认访问的是OpenAI的官方域名 `https://api.openai.com/v1`。DeepSeek的Key当然不认OpenAI的API端点，直接给你返回401认证错误或者404 not found。

然后你一查，发现DeepSeek有自己的API地址，但官网文档里写的又是：

https://api.deepseek.com/v1

是不是有点绕？别担心，其实只需要记住一个核心原则：**不同的API服务商，使用的是不同的baseurl，改Key不改baseurl，等于白改。**

### 第三步：不用SDK，直接发HTTP请求也行

有些同学更喜欢用requests库，一样正确配置baseurl：
python
import requests

url = "https://www.yunwuai.cc/v1/chat/completions" # 注意完整网址

payload = {
    "model": "deepseek-chat",
    "messages": [{"role": "user", "content": "你好"}]
}
headers = {
    "Authorization": "Bearer 你的Key",
    "Content-Type": "application/json"
}

response = requests.post(url, json=payload, headers=headers)
print(response.json()["choices"][0]["message"]["content"])

baseurl必须拼接到完整请求路径里，一个小细节：斜杠不能多也不能少。

### 第四步：验证配置是否生效

跑一下这段代码，如果返回模型的回复而非报错，说明baseurl配对了。常见报错排查：
- **401 Unauthorized**：检查Key是否正确，或者baseurl是否属于这个Key对应的服务商（比如云雾AI的Key要搭配云雾AI的baseurl）
- **404 Not Found**：baseurl写错了，检查末尾是v1还是v2，有没有多/少/
- **Timeout**：网络问题，国内环境推荐使用云雾AI的国内直达baseurl，避免代理

## 关于baseurl，还有哪些坑？

### 坑1：公网地址与内网地址混淆
有些人自建了模型服务，本地用localhost，部署到服务器忘了改成公网IP，也是baseurl写错。

云雾AI兼容OpenAI格式，所有模型统一了公式。但有些服务商给每个模型不同的端点路径，需要单独配置。

### 坑3：拷贝baseurl时漏了“v1”
很多人复制时丢掉了最后的 `/v1` 部分，导致服务找不到。
